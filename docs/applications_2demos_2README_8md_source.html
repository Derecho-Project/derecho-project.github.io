4 ## 1. Simple Data Distribution Service (DDS)
5 This demo shows how to build a simplified [DDS](https://en.wikipedia.org/wiki/Data_Distribution_Service) system based on the Derecho ObjectStore service. Each process in a distributed application can start a number of producers and consumers on any topics. The DDS is responsible for routing the message from the producers to the consumers on the same topics.
6 
7 ### Build the DDS Demo
8 This demo is incorporated in the Derecho building system. So, it is built together with the whole derecho project. To speed up, you can also do
9 ```
10 make dds_demo
11 ```
12 after `cmake`. The binary can be found at `build/applications/demos/dds_demo`.
13 
14 ### Configure and Run
15 The demo read configurations from the main configuration file (Please refer to "Configuring Core Derecho" section in the [derecho document](https://github.com/Derecho-Project/derecho-unified) for more details). The relevant options of DDS are
16 * `DDS_DEMO/producer/topics`: the topics on which a node publishes messages.
17 * `DDS_DEMO/producer/message_freq_hz`: the message rate of the producers.
18 * `DDS_DEMO/consumer/topics`: the topics on which a node subscribes to.
19 Since the DDS Demo rely on [Derecho Plain-old-data Store(dPods)](https://github.com/Derecho-Project/derecho-unified/tree/master/objectstore), we need to provide configurations for dPods as following:
20 ```
21 [OBJECTSTORE]
22 replicas = 0-2
23 persisted = false
24 logged = false
25 ```
26 The most important option in `OBJECTSTORE` section is `replicas`, which designates a set of nodes by IDs to keep the data, and hence be able to create consumers. (Yes, only the replica nodes can start consumers. This is a limitation we are going to remove later in the object store.) The `persisted` and `logged` options need to be set to `false` so far (to be supported later.)
27 
28 Let's assume we have three nodes A(ID=0), B(ID=1), and C(ID=2). A publishes to `topic1`, B publishes to `topic2`, and C publishes to `topic3`; and they send a message per second. A, B, and C are all subscribe to all three topics. Then A's confiugration should look like this:
29 ```
30 # core derecho configurations
31 ...
32 [OBJECTSTORE]
33 replicas = 0-2
34 persisted = false
35 logged = false
36 [DDS_DEMO/producer]
37 topics = topic1
38 message_freq_hz = 1.0
39 [DDS_DEMO/consumer]
40 topics = topic1,topic2,topic3
41 ```
42 B's configuration:
43 ```
44 # core derecho configurations
45 ...
46 [OBJECTSTORE]
47 replicas = 0-2
48 persisted = false
49 logged = false
50 [DDS_DEMO/producer]
51 topics = topic2
52 message_freq_hz = 1.0
53 [DDS_DEMO/consumer]
54 topics = topic1,topic2,topic3
55 ```
56 C's configuration:
57 ```
58 # core derecho configurations
59 ...
60 [OBJECTSTORE]
61 replicas = 0-2
62 persisted = false
63 logged = false
64 [DDS_DEMO/producer]
65 topics = topic3
66 message_freq_hz = 1.0
67 [DDS_DEMO/consumer]
68 topics = topic1,topic2,topic3
69 ```
70 Then, you can run `dds_demo` on nodes A, B, and C. The demo will print the message produced and consumed as following:
71 ```
72 producer@topic1 msg-1
73 consumer@topic1 msg-1
74 consumer@topic2 msg-1
75 consumer@topic3 msg-1
76 ...
77 ```
78 ### Future Features
79 This is a simple demo to be extended with more features we plan to support
80 * Message persistence
81 * Logged messages that can be retrieved with temporal queries
82 * External consumers that need not to be an object store replica
83 * Pointer to large message to avoid unnecessary data transfer
